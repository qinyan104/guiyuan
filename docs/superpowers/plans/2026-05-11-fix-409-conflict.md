# Objective
Fix the 409 Conflict error that occurs when users save modifications to the archive properties ("修缮档案属性").

# Background & Motivation
Users reported getting a `409 Conflict` error when trying to save modifications to archive properties. Investigation revealed two root causes related to the `revision` property (concurrency control):
1. The `PublicationService.listPublications` method fails to include the `revision` field in its response. As a result, the frontend uses an `undefined`/`null` revision (-1L on the server) when editing metadata immediately after listing, causing it to conflict with the actual server version (e.g., 0L).
2. The JPA `@Version` annotation automatically increments the version field upon flush. However, in methods like `updatePublicationMetadata`, `updatePublication`, and `updatePerson`, `publicationRepository.save()` is called without a flush before returning `publication.getRevision()`. This causes the backend to return the old version number to the frontend, leading to future 409 Conflicts upon subsequent edits.

# Key Files & Context
- `backend/src/main/java/com/genealogy/server/service/PublicationService.java`

# Implementation Steps

1. **Include `revision` in `listPublications`**:
   - In `PublicationService.java` inside the `listPublications` method, map the `revision` from the `publication` entity to the result map.
   - Example: Add `result.put("revision", publication.getRevision());` to the mapped values.

2. **Use `saveAndFlush` for publication updates**:
   - In `updatePublicationMetadata()`, replace `publicationRepository.save(publication)` with `publicationRepository.saveAndFlush(publication)`.
   - In `updatePublication()`, replace `publicationRepository.save(publication)` with `publicationRepository.saveAndFlush(publication)`.
   - In `updatePerson()`, replace `publicationRepository.save(publication)` with `publicationRepository.saveAndFlush(publication)`.
   - This ensures that the newly incremented `revision` is populated in the entity before `return publication.getRevision()` is called.

# Verification & Testing
1. In the frontend, load the publications list and verify that the `revision` field is populated correctly in the network response.
2. Edit the metadata of a publication ("修缮档案属性") and click "封装保存" (Save). Verify that the request uses the correct `expectedRevision` and succeeds with a 200 OK.
3. Edit the metadata a second time consecutively to ensure the `newRevision` returned from the first update was correctly incremented and passed along, preventing any 409 Conflicts.
