# 分支挂载与物理合并设计说明

状态：已按“安全挂载管理”方案收敛。本文件用于说明当前实际落地范围，并取代早期“主画布直接递归展开远程分支”的设想。

## 当前结论

分支协作被拆成两种动作：

- `挂载`：只建立关联关系，不复制数据
- `物理合并`：把目标族谱当前快照复制进主族谱

当前实现明确不做：

- 主画布自动展开远程分支
- 将远程分支混入可编辑、可自动保存的数据流

这样做的原因是，编辑器存在保存链路；如果直接把远程数据灌进主族谱工作区，就容易出现“未点合并却误写入主族谱”的高风险问题。

## 已落地范围

### 1. 挂载管理

- 支持将普通人物升级为挂载点
- 支持断开挂载关联
- 侧边面板展示目标族谱元数据
- 当前提示文案已与安全方案保持一致

### 2. 权限模型

- 执行挂载、断开、物理合并的人，必须是主族谱 `OWNER`
- 对目标族谱，执行人至少要有 `VIEWER`
- 平台 `ADMIN / SUPER_ADMIN` 不自动绕过族谱级授权

### 3. 安全数据流

- `PublicationService.loadPublication` 只返回当前族谱的本地可编辑数据
- 挂载点信息以元数据形式附带返回
- 普通保存不会把远程分支误写回主族谱

### 4. 物理合并

- 以挂载点对应的目标族谱根人物为入口
- 复制人物、家庭和照片数据
- 做 `personId / familyId` 唯一化与重映射
- 支持递归整合子分支挂载
- 合并后清除挂载状态，主族谱与原分支彻底解耦

## 不再采用的旧方案

以下思路已明确放弃，不应再作为当前实现目标：

- 主画布上的 `[+] / [-]` 远程分支展开
- 通过递归跨谱加载结果直接驱动画布编辑态
- 把“挂载”做成“看起来已经并入总谱”的交互

如果后续确实需要“展开查看远程分支”的体验，应另做一条只读预览链路，而不是复用当前可回写的编辑数据流。

1. 挂载、断开、物理合并主链路可用
2. 权限拦截符合主族谱 `OWNER` + 目标族谱可读的规则
3. 普通编辑与自动保存不再污染主族谱数据
4. 文档、面板提示、开发说明全部与当前实现一致
