# 2026-05-04 后台谱馆化视觉重构设计

## 背景

当前后台主壳由 `frontend/src/views/AdminLayout.vue` 提供，左侧包含四个主入口：

- `dashboard` 首页
- `publications` 族谱管理
- `admin-users` 用户管理
- `admin-logs` 操作日志

现状问题不是单页质量不足，而是整套后台缺少统一视觉语言：

- `PublicationListView.vue` 已有较强的展陈感和编辑性版式。
- `DashboardView.vue`、`AdminUsersView.vue`、`AuditLogView.vue` 仍接近常规管理后台。
- `AdminLayout.vue` 的侧栏与顶栏是第三套语言，和内容页没有形成完整系统。
- 若继续逐页独立优化，最终会得到四个风格相近但仍不统一的页面。

本设计将后台重新定义为一套“当代谱馆目录系统”，目标是在不牺牲管理效率的前提下，统一左侧栏、顶栏和四个核心页面的视觉与版式。

## 已确认决策

- 视觉方向：`典雅谱牒风`
- 信息密度：`平衡型`
- 改造范围：`左侧栏 + 顶栏 + 四个页面内容区`
- 推荐方案：`馆藏后台融合式`

该方案的原则是：

- 首页与族谱管理承担更强的品牌气质与内容展陈。
- 用户管理与操作日志保留后台效率，但转译为“名录册 / 流水簿”语义。
- 所有页面共享同一套布局栅格、色彩、字体、边框、状态与弹窗系统。

## 目标

1. 建立统一的后台壳层视觉系统，使四个页面切换时有稳定、连续的产品感。
2. 让后台气质与“族谱 / 谱牒 / 档案”主题一致，而不是套用通用 SaaS 控台风格。
3. 保持用户管理、操作日志等高密度页面的可读性与操作效率。
4. 在尽量不改动数据流与业务逻辑的前提下完成一次以视图层为主的重构。

## 非目标

- 不重写后台路由结构。
- 不改变现有 API 协议。
- 不在本轮引入新的后台业务功能。
- 不将后台全部做成纯展陈页；管理效率优先级仍高于视觉表演。

## 设计总述

后台整体采用“谱馆后台”母题：

- 基底：暖纸色、宣纸白、淡墨灰、低饱和金褐、克制朱砂。
- 结构：细边框、浅分层、弱阴影、稳定栅格，不使用厚重卡片阴影和高饱和渐变。
- 字体策略：标题偏书卷感 serif，正文、表格与交互仍使用清晰 sans。
- 动效策略：仅保留微抬升、边线显影、分区淡入等低强度反馈。

视觉关键词：

- 卷首
- 目录签条
- 馆藏卡
- 名录册
- 流水簿
- 馆务批注

## 信息架构与实现边界

### 路由与页面边界

本次设计覆盖以下视图：

- `frontend/src/views/AdminLayout.vue`
- `frontend/src/views/DashboardView.vue`
- `frontend/src/views/PublicationListView.vue`
- `frontend/src/views/AdminUsersView.vue`
- `frontend/src/views/AuditLogView.vue`

`SettingsView.vue` 不属于本轮核心页面，但由于共用后台壳层，应在实现阶段被动继承新的侧栏、顶栏与基础组件样式。

### 数据流边界

本次重构以视图层和样式层为主，不调整现有数据流：

- `DashboardView.vue` 继续使用现有 publication/user summary 请求。
- `PublicationListView.vue` 继续使用现有创建、编辑、删除、模板创建流程。
- `AdminUsersView.vue` 继续使用现有用户列表、创建、重置密码、删号、改角色流程。
- `AuditLogView.vue` 继续使用现有日志请求与错误分支。

允许的“轻逻辑增强”仅限于为了视觉呈现而增加的前端衍生数据，例如：

- 今日记录数
- 某角色数量摘要
- 最近更新时间文案
- 风险操作计数

这些都应由当前已获取的数据在前端本地派生，不新增接口依赖。

## 核心设计

### 1. 后台壳层：侧栏 + 顶栏 + 内容容器

#### 左侧栏

左侧栏重构为“馆藏目录栏”，承担全局定位功能。

要求：

- 宽度稳定在 `232px-248px` 区间。
- 背景色略深于主内容区，形成纸页叠层关系。
- 保留品牌区、主导航区、工具区三段式结构。
- 导航项不再使用整块高饱和激活背景，改为：
  - 左侧细竖条
  - 浅米金底
  - 深墨文字
- hover 态只做轻底色与边线反馈，避免“按钮感”过重。
- 可为导航项增加弱化数量信息或状态点，但不能抢主文本层级。

品牌区应重构为：

- 馆印式图形
- 系统名称
- 一行很小的副题

工具区包含：

- 设置
- 退出

工具区与主导航之间必须有明确分隔，避免与内容导航混读。

#### 顶栏

顶栏重构为“卷首眉栏”，其职责不是放工具，而是提供当前页面定位。

要求：

- 左侧显示当前页标题与层级归属，例如：
  - `后台总览 / 工作台`
  - `谱牒馆藏 / 族谱管理`
  - `系统治理 / 用户管理`
  - `系统治理 / 操作日志`
- 右侧保留主题切换、当前用户、退出，但视觉权重弱于左侧标题。
- 顶栏使用半透明暖白底和极细底边线，避免强玻璃拟态。

#### 内容容器

所有后台页统一使用同一栅格容器：

- 一致的最大内容宽度
- 一致的左右边距
- 一致的页首与主内容区间距

避免出现某些页面宽松、某些页面拥挤、某些页面漂浮在中央的情况。

### 2. 页面一：首页（DashboardView）

首页从“统计卡片页”重构为“卷首总览页”。

#### 新结构

- 页首标题区：标题、简短说明、可选的系统状态提示
- 上半区：
  - 一块主叙事卡，展示馆藏概况
  - 两到三块次级摘要卡，展示用户、备份、快捷操作
- 下半区：
  - 最近编研卷册列表
  - 可选的今日提醒或最近动作摘要

#### 视觉原则

- 主叙事卡优先级高于统计数字卡。
- 数字不再简单四等分平铺，而是形成“主次层级”。
- 最近族谱列表更像“最近编研卷册”，每条应有标题、副题、更新时间与轻量状态提示。

#### 保留能力

- 跳转到族谱管理
- 跳转到用户管理
- 备份入口
- 打开最近族谱

### 3. 页面二：族谱管理（PublicationListView）

该页作为整套后台的视觉基准页，保留其展陈气质，但需要收敛并去混搭。

#### 保留

- 大标题与卷册式首屏节奏
- 模板区与私人档案区的分层
- 卡片化档案展示

#### 调整

- 减少当前过多的英文标签和全大写文案，统一为中文主导的“谱馆语气”。
- 模板区明确为“精选谱式 / 示例卷册”，私人档案区明确为“馆藏目录 / 我的编研卷册”。
- 卡片操作按钮、标签、弹窗标题与按钮语气统一。
- 删除确认层从技术式 overlay 调整为更克制的“危险批注层”。

#### 版式要求

- 模板区与私人档案区之间的节奏差异要清晰。
- 卡片信息层级应优先标题、次级副题、再次级元数据，不允许同级信息过多。
- 卡片操作区应弱化，避免压过卷册本身。

### 4. 页面三：用户管理（AdminUsersView）

该页定位为“名录册 / 人员簿”，保留后台效率，但脱离通用数据表风格。

#### 新结构

- 页首标题区：页面标题、管理说明、角色分布摘要
- 筛选区：角色分类签条
- 新建区：可展开的“新增条目”面板
- 主体区：名录式列表或表格

#### 视觉原则

- 角色筛选不再像传统 tab，而应更像目录标签或分类签条。
- 新建用户表单不能像临时拼接在页面中部，应具备独立面板语义。
- 表格改为“档案名录”视觉：
  - 更舒展的行高
  - 更克制的表头
  - 更细的分隔
  - 更轻的角色徽签

#### 交互要求

- 行内操作优先轻量按钮，不使用强烈色块。
- 超级管理员行可保留保护态，但要用统一的“馆藏保护项”语言，而不是突兀警示。
- 重置密码、删除确认应统一到共享对话框体系。

### 5. 页面四：操作日志（AuditLogView）

该页定位为“馆务记录 / 行为流水簿”。

#### 新结构

- 页首标题区：标题、说明
- 摘要带：今日记录数、高风险操作数、最近备份等
- 主体区：按时间阅读的流水列表

#### 视觉原则

- 日志不再只是一张僵硬表格，而是更接近“时间序列档案”。
- 时间、操作、详情之间需要形成明确阅读顺序。
- 高风险操作用统一的朱砂风险语言标识，但不铺满整页红色。

#### 错误态要求

- 当前接口缺失提示应改为“馆务提示条”而非原始技术报错风格。
- 当日志为空时，应给出合理空态说明，而非只显示“暂无记录”。

### 6. 统一组件与视觉规则

本轮应沉淀以下共用组件或样式语义：

- 卷首标题块
- 摘要卡
- 目录签条
- 馆藏卡片
- 名录表格
- 流水记录行
- 徽签标签
- 轻操作按钮
- 危险确认对话框
- 提示条
- 骨架屏 / 加载占位

要求：

- 所有页面优先复用这套语义，而不是每页单独定义按钮和面板。
- 共享规则应尽量下沉到 `AdminLayout.vue`、共享 CSS token、或可复用组件中。

### 7. 文案与语气统一

当前部分页面存在中英混搭、全部大写、以及疑似编码异常带来的观感问题。

本轮必须同步统一文案策略：

- 中文为主，必要时保留很少量英文副题。
- 同类动作命名保持一致，例如“新建”“保存”“删除”“取消”“重置密码”。
- 风险动作与提示条使用同一套措辞。
- 页面标题、摘要卡标题、弹窗标题不要混用展陈文案与技术文案。

实现前应先解决编码/文案渲染问题，否则视觉重构结果会被破坏。

### 8. 状态设计

四页统一以下状态表现：

#### 加载态

- 使用骨架屏或低干扰占位块。
- 禁止仅保留一行“加载中”文本作为最终交付状态。

#### 空状态

- 提供主题化标题和一句说明。
- 给出明确的下一步动作，例如：
  - 去创建族谱
  - 去新增用户
  - 当前暂无操作记录

#### 错误态

- 使用统一提示条组件。
- 技术细节只在必要时展示，不应以接口路径为主文案。

#### 危险态

- 删除、清除、重置密码、备份等动作统一使用朱砂风险色与统一确认层级。

### 9. 响应式策略

后台以桌面端优先，但必须兼顾平板，并避免手机端完全失效。

#### 桌面端

- 完整侧栏
- 完整顶栏
- 多列布局

#### 平板端

- 侧栏允许折叠
- 摘要区从多列收敛为两列或单列
- 宽表格允许局部重排

#### 手机端

- 侧栏改抽屉
- 用户管理与日志优先转为卡片化纵向字段堆叠
- 不强行保留完整表格列结构

## 实施原则

1. 先统一壳层，再统一共享 token/组件，再逐页改造。
2. 先解决文案编码问题，再做最终视觉 polish。
3. 优先复用已有结构和数据流，避免视觉重构演变为业务重写。
4. `PublicationListView.vue` 作为视觉锚点，但需要做“回归谱馆语境”的收敛，而不是简单复制其现状。

## 风险与规避

### 风险 1：过度展陈化，降低后台效率

规避：

- 只让首页和族谱管理承担较强的视觉表达。
- 用户管理和日志页保留清晰的操作路径与阅读顺序。

### 风险 2：页面风格统一失败

规避：

- 优先设计共享组件语义与 token。
- 禁止逐页自由发挥。

### 风险 3：视觉完成后仍显凌乱

规避：

- 同步处理文案、编码、字重、间距、状态与弹窗语言。
- 统一标题体系与容器尺度。

## 验收标准

满足以下条件时，视为本轮设计落地成功：

1. 左侧栏、顶栏、四个页面看起来属于同一个产品，而不是四套页面拼接。
2. 用户在四页之间切换时，能感知一致的标题、间距、层级与状态语言。
3. 首页与族谱管理具有明显的“谱馆”气质。
4. 用户管理与操作日志仍然可高效完成筛选、查看、删除、重置等任务。
5. 所有弹窗、空态、加载态、错误态遵循同一套设计规则。
6. 不引入新的接口依赖，不破坏现有主要业务流程。

## 测试与验证

本轮以视觉重构为主，验证重点应放在以下方面：

### 视觉一致性检查

- 四页页首结构是否统一
- 导航激活态是否统一
- 弹窗、标签、提示条是否统一
- 字体、字号、边框、阴影是否统一

### 交互回归检查

- 首页各快捷入口是否仍可跳转
- 族谱创建、编辑、删除、模板创建是否不受影响
- 用户创建、改角色、重置密码、删除是否不受影响
- 日志加载、空态、错误态是否仍正确显示

### 响应式检查

- 宽屏桌面
- 常见笔记本宽度
- 平板宽度
- 窄屏移动端的基础可用性

### 文案检查

- 是否仍有乱码
- 是否仍有不必要的英文全大写标签
- 是否存在同义动作多种写法

## 后续衔接

2. 共享 token / 共享组件抽取
3. 首页改造
4. 族谱管理页收敛
5. 用户管理页重构
6. 操作日志页重构
