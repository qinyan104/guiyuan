# 本地闭环版族谱出谱工作台设计

## 1. 背景

当前项目已经具备以下原型能力：

- 人物卡片排版与谱图绘制
- 人物信息编辑
- 配偶、父母、子女关系编辑
- 分支聚焦
- 撤销与重做
- 本地草稿恢复
- SVG 导出与打印

现阶段的主要问题不是“缺少框架”，而是“业务闭环尚未完成”。项目当前仍主要依赖示例数据，核心编辑逻辑集中在单个大文件中，数据导入、结构校验、关系合法性约束、错误反馈和正式交付流程还不完整。

因此，下一阶段不继续扩展空泛的系统骨架，而是把现有原型收敛成一个真正可用的单机版族谱编辑与出谱工具。

## 2. 产品定位

本阶段将项目定位为：

**本地单机版族谱编辑与出谱工作台**

它的目标用户是需要整理家族人物关系、调整谱图排版、导出印刷版或电子版族谱页面的单人操作者。

本阶段的核心价值是：

- 能导入一份正式族谱数据
- 能安全地编辑人物与关系
- 能稳定保存与恢复草稿
- 能导出可交付的出谱结果

## 3. 阶段目标

本阶段完成后，项目应满足以下结果：

1. 用户可以从本地导入 JSON 族谱数据并开始编辑。
2. 系统能够在导入时识别结构错误、引用错误和明显的业务冲突。
3. 所有人物与关系编辑动作都通过统一的操作层执行，而不是散落在界面事件中直接修改数据。
4. 草稿可以自动保存、恢复、重置，并具备清晰的数据版本边界。
5. 用户可以导出 JSON 数据、导出 SVG 图稿，并发起打印。
6. 核心业务规则有最小但有效的测试覆盖，后续扩展不会轻易破坏关系结构。

## 4. 明确不做

为避免范围失控，本阶段明确不做以下内容：

- 登录与权限体系
- 后端接口
- 数据库建模与持久化服务
- 多人协作
- 审批流
- 复杂运营后台
- 与微信、短信、邮件等外部系统集成

这些能力只有在“本地单机版工作台”稳定后才有讨论价值。

## 5. 当前代码现状

当前代码结构以单页应用为主，关键问题如下：

- `src/App.vue` 承担了过多职责，包括数据状态、编辑逻辑、历史记录、草稿持久化、导出打印和 UI 组织。
- `src/data/sampleFamily.ts` 仍是主要数据来源，缺少正式导入入口。
- `src/types/family.ts` 仅覆盖基础排版数据结构，尚未表达导入校验、草稿元信息、错误类型等正式领域模型。
- `src/lib/layout.ts` 已具备独立的排版职责，这是当前代码中应保留并继续强化的清晰边界。

这意味着下一阶段最重要的工作不是再补页面壳子，而是建立稳定的数据链路和可维护的代码边界。

## 6. 总体方案

采用“本地闭环优先”的方案，统一走以下数据流：

`导入 JSON -> 结构校验 -> 归一化 -> 编辑 -> 自动保存 -> 导出 JSON / SVG / 打印`

该方案的设计原则如下：

- 先保证单机可用，再考虑服务端扩展。
- 先保证关系正确，再做更多视觉打磨。
- 先让编辑动作经过统一业务层，再谈更复杂的功能。
- 先把大文件拆出清晰边界，再继续加功能。

选择该方案的原因：

- 它与当前已有原型能力连续，不需要推倒重来。
- 它能最短路径把项目从“演示原型”推进到“可交付工具”。
- 它保留了后续接后端的可能性，因为数据模型、校验层和操作层本身就是将来服务端化的基础。

## 7. 模块边界设计

下一阶段不做全面重构，但要进行受控拆分。推荐模块边界如下。

### 7.1 编辑模块

目录建议：`src/features/editor/`

职责：

- 当前选中人物状态
- 人物基础信息编辑
- 新增配偶、子女、父母
- 删除人物
- 分支聚焦
- 子女顺序调整

约束：

- 不直接负责持久化
- 不直接负责布局计算
- 所有关系变更通过统一操作函数完成

### 7.2 历史记录模块

目录建议：`src/features/history/`

职责：

- 快照生成
- 撤销
- 重做
- 历史标签归类
- 历史数量限制

约束：

- 只处理编辑状态的时间维度变化
- 不承担业务校验本身

### 7.3 持久化模块

目录建议：`src/features/persistence/`

职责：

- 本地草稿保存与恢复
- 导入 JSON
- 导出 JSON
- 草稿版本号管理
- 兼容旧草稿结构

约束：

- 持久化前后都要走类型与结构校验
- 不允许未归一化数据直接进入编辑态

### 7.4 校验模块

目录建议：`src/features/validation/`

职责：

- 数据结构校验
- 引用完整性校验
- 家庭关系合法性校验
- 编辑操作前置校验
- 用户可读错误信息生成

约束：

- 校验结果必须可供界面直接展示
- 校验失败时不得产生部分写入状态

### 7.5 排版模块

保留 `src/lib/layout.ts` 作为独立排版层。

职责：

- 根据标准化后的出版数据与排版配置计算卡片与连线布局

约束：

- 不关心本地存储
- 不关心 UI 面板
- 不直接处理导入错误

### 7.6 领域类型模块

保留并扩展 `src/types/family.ts`。

新增表达能力应包括：

- 导入数据版本
- 草稿元信息
- 校验错误类型
- 标准化后的编辑数据结构
- 关系操作结果

## 8. 数据模型设计

本阶段继续采用“人物 + 家庭单元”的表达方式，不引入更复杂的图数据库式结构。

原因：

- 当前布局算法已经围绕 `people` 与 `families` 组织
- 当前 UI 也已经以“人物编辑”和“家庭分支”为核心
- 该结构足以支撑本阶段的单机编辑和出谱目标

标准数据应至少包含以下概念：

### 8.1 人物

- `id`
- `name`
- `gender`
- `birth`
- `death`
- `age`
- `note`

### 8.2 家庭单元

- `id`
- `adults`
- `children`

### 8.3 出谱文档

- `title`
- `subtitle`
- `focusFamilyId`
- `people`
- `families`

### 8.4 设置

- 纸张尺寸
- 卡片宽度
- 代际间距
- 兄弟姐妹间距
- 配偶间距
- 字号缩放
- 画布缩放
- 显示死亡信息开关
- 显示年龄开关
- 显示备注开关
- 内边距

### 8.5 草稿包

建议新增一个顶层草稿概念，用于本地持久化与导入导出：

- `version`
- `savedAt`
- `publication`
- `settings`

这样可以把“正式编辑数据”和“本地草稿元信息”分开，后续兼容升级也更清晰。

## 9. 数据导入与导出

### 9.1 导入

导入流程必须严格分层：

1. 读取本地 JSON 文本
2. 解析 JSON
3. 校验顶层结构是否符合草稿包或出版数据格式
4. 校验人物与家庭引用是否完整
5. 归一化缺省字段
6. 进入编辑状态

导入失败时必须明确告知失败原因，例如：

- JSON 格式错误
- 缺少 `people`
- 缺少 `families`
- `focusFamilyId` 指向不存在家庭
- 家庭中的人物 ID 不存在
- 重复或冲突关系

### 9.2 导出 JSON

导出 JSON 是本阶段必须补齐的正式能力，用于：

- 数据备份
- 跨设备迁移
- 正式资料交接
- 后续接后端时作为临时交换格式

导出内容应优先采用“草稿包”结构，而不是只导出裸 `publication`，这样可以保留必要配置与版本信息。

### 9.3 导出 SVG 与打印

当前已有 SVG 导出与打印能力，应保持能力不回退，并确保它们依赖的是经过校验和归一化后的有效布局数据。

如果当前数据无法生成有效布局，应阻止导出并提示原因，而不是导出损坏内容。

## 10. 业务规则设计

这一部分是本阶段的核心。系统必须优先保证关系正确，而不是优先响应按钮点击。

### 10.1 统一关系操作层

新增、删除和调整关系时，不允许在视图事件中直接随意修改 `publication.people` 或 `publication.families`。所有操作必须走统一函数，例如：

- 新增人物
- 新增配偶关系
- 新增父母关系
- 新增子女关系
- 删除人物
- 移除配偶关系
- 移除父母关系
- 调整子女顺序
- 切换焦点家族

统一操作层的职责是：

- 检查前置条件
- 保持数据双向一致
- 处理级联清理
- 返回成功或失败结果

### 10.2 必须阻止的问题

系统至少要阻止以下问题进入状态树：

- 家庭引用不存在的人物
- 人物同时以冲突方式存在于多个不合理家庭关系中
- 焦点家庭不存在
- 删除人物后仍残留脏引用
- 建立明显循环的父母关系
- 对同一关系重复创建相同连接

### 10.3 删除规则

删除人物时，系统必须同步清理：

- 该人物在 `people` 中的记录
- 其作为配偶时所在家庭的引用
- 其作为子女时在父母家庭中的引用
- 因清理后变为空壳的家庭单元
- 删除后失效的焦点家庭引用

删除前必须提供明确影响摘要，至少告诉用户：

- 会移除哪些关系
- 是否会导致某些家庭被清理
- 焦点分支是否会发生切换

### 10.4 排序规则

子女顺序会直接影响出谱左右布局，因此“调整顺序”不是纯 UI 操作，而是正式数据编辑动作，应进入历史记录并写入草稿。

## 11. 界面行为设计

本阶段不做大规模视觉改版，继续沿用当前工作台交互，但要让行为更稳定。

### 11.1 保留的核心体验

- 中央画布编辑
- 右侧人物编辑抽屉
- 关系操作按钮
- 迷你地图
- 撤销与重做
- 概览面板

### 11.2 需要新增的界面能力

- 导入 JSON 入口
- 导出 JSON 入口
- 结构校验失败的可读提示
- 空草稿状态提示
- 导入覆盖当前草稿前的确认
- 草稿版本不兼容时的明确提示

### 11.3 错误提示原则

- 错误提示必须具体，不能只显示“操作失败”
- 文案要指出失败原因和受影响对象
- 如果是导入失败，尽量指出错误字段或错误引用
- 如果是关系编辑失败，尽量指出冲突关系

## 12. 错误处理

错误处理分三类：

### 12.1 输入错误

例如：

- 非法 JSON
- 表单缺少必要人物名称
- 无法识别的性别值

处理方式：

- 阻止提交
- 给出就近提示

### 12.2 结构错误

例如：

- 家庭引用了不存在人物
- 焦点家庭不存在
- 导入数据缺少必要字段

处理方式：

- 拒绝进入编辑状态
- 给出明确错误描述

### 12.3 业务冲突

例如：

- 重复建立关系
- 删除后导致数据断链
- 形成明显循环关系

处理方式：

- 阻止操作
- 保持原状态不变
- 给出冲突说明

## 13. 测试策略

本阶段测试目标不是追求高覆盖率，而是保护核心关系逻辑和数据闭环。

测试重点如下：

### 13.1 校验层测试

- 合法导入数据可通过
- 缺字段数据被拒绝
- 不存在人物引用被拒绝
- 不存在焦点家庭被拒绝

### 13.2 关系操作测试

- 新增配偶后家庭结构正确
- 新增子女后父母家庭与子女人物引用一致
- 新增父母后能建立正确上层家庭
- 删除人物后脏引用被同步清理
- 调整子女顺序后顺序被正确持久化

### 13.3 持久化测试

- 保存草稿后可恢复
- 旧版本草稿能被识别并合理处理
- 导出 JSON 后再次导入结果一致

### 13.4 排版结果测试

- 合法数据能够生成布局
- 切换焦点家庭后仍能生成有效布局
- 空或异常数据不会错误触发导出

## 14. 迭代顺序

本阶段按以下顺序实施：

### 第一包：数据导入导出与草稿版本

目标：

- 让项目脱离示例数据依赖
- 建立正式数据入口和出口

交付结果：

- 导入 JSON
- 导出 JSON
- 草稿版本字段
- 恢复与重置逻辑清晰

### 第二包：关系规则校验与统一操作层

目标：

- 让所有编辑动作变得可靠

交付结果：

- 统一关系操作函数
- 校验结果对象
- 删除与关系清理规则固化

### 第三包：`App.vue` 受控拆分

目标：

- 降低单文件复杂度
- 为后续扩展留下维护空间

交付结果：

- 编辑逻辑拆分
- 历史逻辑拆分
- 持久化逻辑拆分
- 校验逻辑拆分

### 第四包：错误提示、空状态与交付前打磨

目标：

- 让工具真正可交付给非开发者使用

交付结果：

- 明确错误提示
- 空状态提示
- 导入覆盖确认
- 失败阻断机制

## 15. 后续扩展接口

本阶段虽然不做后端，但设计必须保留后续扩展空间。

应保留的扩展可能性：

- 把本地导入导出替换为服务端保存
- 把统一关系操作层复用到接口层
- 把校验规则迁移到共享领域层
- 在草稿包版本基础上做迁移脚本

这也是为什么本阶段先补“数据模型、校验、操作层”，而不是直接接数据库。

## 16. 成功标准

本阶段完成后，可以用以下标准判断是否达到目标：

- 不依赖示例数据也能正常开始编辑
- 任意一次人物和关系编辑不会轻易破坏数据结构
- 删除人物后不会遗留脏关系
- 刷新页面后草稿仍可恢复
- 导出的 JSON 能再次成功导入
- SVG 导出和打印只在有效布局下工作
- 主入口文件职责明显下降，后续扩展不再必须继续堆进单个文件

## 17. 实施结论

下一阶段不应继续泛化“管理系统”范围，而应先把当前项目完成为一个稳定的、本地闭环的族谱编辑与出谱工具。

这是当前代码基础上最短、最稳、最有产出的路线。等这一步完成后，再决定是否继续向“登录 + 后端 + 数据库”的完整管理系统演进，届时才是合理的扩张时点。
