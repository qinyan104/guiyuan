# 分支挂载管理操作手册

适用范围：当前“安全挂载管理”方案。

一句话原则：
- `挂载` = 建立关联，不复制数据
- `物理合并` = 复制当前快照，永久写入主族谱

## 角色分工

平台角色：
- `SUPER_ADMIN`：负责用户、审计、备份、系统层面协调
- `ADMIN`：负责后台管理，但不自动拥有任意族谱的编辑权

族谱角色：
- `OWNER`：可编辑、分配权限、创建分享、管理挂载、执行物理合并
- `EDITOR`：可编辑本谱，但不能管理挂载和权限
- `VIEWER`：只读

关键约束：
- 执行 `挂载 / 断开 / 物理合并` 的人，必须是主族谱的 `OWNER`
- 对目标分支族谱，执行人至少要有 `VIEWER` 权限
- `SUPER_ADMIN` 不会自动跳过上述族谱权限模型

## 推荐组织方式

建议按“两层负责制”协作：

1. 总谱设置 `1-2` 名总协调人，赋予总谱 `OWNER`
2. 每个分支族谱各自设置负责人，赋予该分支 `OWNER`
3. 总协调人如果只需要挂载或合并某个分支，对该分支赋予 `VIEWER` 即可
4. 只有确实需要跨谱代修时，才把总协调人提升为分支 `EDITOR` 或 `OWNER`

## 标准操作流程

### 1. 协作准备

上线前先完成三件事：
- 明确哪一本是总谱，哪些是分支谱
- 明确每本族谱的 `OWNER / EDITOR / VIEWER`
- 完成一次数据库备份

### 2. 建立挂载

总协调人进入总谱，在目标人物上建立挂载关联：
- 选择要关联的分支族谱
- 建立挂载点
- 核对侧边面板显示的目标族谱名称是否正确

注意：
- 挂载后主画布不会自动展开远程分支
- 普通保存不会把对方数据写进来

### 3. 日常协作

日常修谱建议遵守以下规则：
- 分支负责人只在自己的分支谱内维护
- 总谱负责人只维护总谱本体和挂载关系
- 不把“物理合并”当作日常同步手段

### 4. 收官前冻结

准备把某个分支正式纳入总谱前：
- 提前通知该分支负责人暂停编辑
- 确认最近一次修改已经保存
- 再做一次数据库备份

### 5. 执行物理合并

由总谱 `OWNER` 执行：
- 找到挂载点
- 点击 `执行物理合并`
- 等待系统完成
- 合并完成后刷新总谱并抽查关键人物关系

合并后含义：
- 目标分支当前快照已经复制进总谱
- 原挂载点会转为普通人物节点
- 后续再修改原分支，不会自动同步回总谱
- 如果物理合并或总谱自动保存撞车，后端会返回 `409`，先让其他编辑会话结束，再刷新重试。

### 6. 审计与归档

- 总谱保存与物理合并现在按谱串行写入；正式合并前，尽量关闭其他正在编辑同一本总谱的标签页。

每次重要操作后，建议检查：
- 审计日志里是否记录了挂载、断开、合并动作
- 是否留存本次合并前的备份
- 是否已经通知相关分支“本轮汇总已完成”

## 不建议的做法

- 不要让所有人直接在总谱里改同一批内容
- 不要未备份就执行物理合并
- 不要把平台管理员当成天然的全谱编辑者
- 不要把挂载误认为“已经并入总谱”

## 适合的节奏

如果你们是多人长期协作，推荐节奏是：
- 平时分支独立维护
- 阶段性检查挂载关系
- 在月末、版本发布前或成书前统一执行物理合并
