# 族谱管理系统未来路线规划

更新日期：2026-04-26

## 1. 文档目的

这份文档用于总结当前项目的阶段判断、未来定位、技术演进思路与分期路线，作为后续持续开发时的指导基线。

它不是需求细节文档，也不是数据库设计稿，而是一份面向中长期的产品与技术路线说明。

## 2. 当前项目判断

结合现有代码结构，可以明确当前项目已经完成了一个有明显特色的作品级原型：

- 前端已经具备较强的族谱可视化编辑能力
- 支持人物编辑、关系维护、画布展示、导入导出、打印与 SVG 输出
- 前端的核心优势在于“可视化体验”和“本地编辑工作流”
- 后端目前主要提供登录注册与文件上传能力
- 族谱核心内容目前仍以本地前端状态和 JSON 结构为主

因此，当前项目更准确的定义不是“完整管理后台”，而是：

**一个以可视化编辑为核心的本地族谱工作台原型。**

这意味着项目已经具备作品价值，但距离产品化和系统化仍有明显演进空间。

## 3. 项目未来定位

本项目未来建议定位为：

**一套可本地部署、视觉优先、面向中文家族场景的家族记忆系统。**

这一定义包含四个核心关键词：

- 可本地部署：数据主权归使用者，支持私有化运行
- 视觉优先：可视化体验不是附属能力，而是产品核心竞争力
- 中文家族场景：围绕族谱、支系、字辈、家族资料等真实使用语境展开
- 家族记忆系统：不只管理谱系图，还要沉淀人物、事件、文档、证据与传承记录

## 3.1 产品意义补充

这个项目的意义不应只被理解为“记录一份族谱”，而应被理解为：

- 为家庭和家族留下来处与脉络
- 为后代保留仍可回看的名字、关系与故事
- 在关系逐渐松散、记忆容易断裂的现实中，提供一种温和的保存方式

因此，本项目不是为了恢复旧式宗族秩序，而是为了在现代生活语境下，保存家庭记忆与来处感。

## 3.2 面向用户的表达方式

从产品表达上，不建议长期把它只定义成“族谱管理系统”。

更适合的表达方式是：

- 家族记忆系统
- 家庭档案整理工具
- 家人关系与故事的本地化整理工具

换句话说，族谱是结构骨架，记忆才是产品真正想承接的内容。

## 4. 核心产品原则

后续所有功能演进，建议尽量围绕以下四条主轴，不轻易偏离：

### 4.1 Local-first

- 离线优先
- 私有部署优先
- 用户对数据有完全控制权
- 系统不应以依赖外部平台为前提

### 4.2 Visual-first

- 谱图展示能力必须持续保持领先
- 交互体验不能退化成普通后台页面
- 关系、结构、历史资料要尽量可视化表达

### 4.2.1 入口轻量化

- 不让用户一上来就面对沉重的“修谱”心智
- 先让用户从家人、照片、故事和关系图进入
- 先降低门槛，再逐步进入深度维护

### 4.3 History-first

- 家族资料要能长期沉淀
- 要重视版本、修订、来源、证据和恢复能力
- 不只是“能录入”，更要“可追溯、可维护、可传承”

### 4.4 AI-assisted

- AI 的价值应放在资料整理、结构化抽取、冲突发现和录入辅助
- 不为了追热点而强行引入 AI
- AI 必须服务于本地化与数据主权，而不是削弱它们

## 5. 总体演进策略

项目不适合一次性重构成“大而全平台”，否则工程压力会迅速失控。

更合理的方式是分三步走：

1. 先把现有作品做成可靠产品
2. 再把产品做成真正的家族资料系统
3. 最后发展出具有差异化价值的平台能力

换句话说，建议路线是：

**作品 -> 产品 -> 系统 -> 平台**

在这个过程中，产品视角上的演进还应补上一条隐含路线：

**族谱工具 -> 家庭记忆产品 -> 家族资料系统**

## 6. 分阶段路线图

## 6.1 第一阶段：1.0 产品化

### 目标

把当前“自己能用的可视化作品”，升级为“别人也能稳定使用的本地产品”。

### 阶段定义

1.0 的核心目标不是功能做大，而是把以下能力做稳：

- 数据可长期保存
- 编辑过程可恢复
- 本地部署可执行
- 用户可以放心把真实家族数据放进去
- 用户第一次接触时不会被“修谱压力”劝退

### 必做方向

- 前端编辑器与后端存储打通
- 族谱数据可创建、打开、保存、另存为
- 用户可登录并持有自己的族谱项目
- 图片与附件可稳定上传和关联
- 导入导出流程规范化
- 基础异常提示和保存失败处理
- 快照恢复或历史版本基础能力
- 本地部署启动文档完善
- 第一屏入口与产品语言更轻量、更易理解
- 支持从“少量家人 + 少量记忆”开始，而不是强迫完整建谱

### 暂不优先

- 复杂角色权限
- 多组织协同
- 审批流
- 大量后台运营页面
- AI OCR 与智能抽取
- 迁徙图、时间轴等高级展示

### 技术建议

1.0 阶段不建议一开始就把族谱模型全面范式化。

可以采用过渡方案：

- 用户表：管理账号
- 族谱项目表：管理一份族谱项目的元信息
- 族谱内容表：保存当前族谱 JSON 内容
- 附件表：保存头像、图片或附属文件信息

这种做法可以先把“服务端保存链路”跑通，降低改造成本。

### 成功标准

如果用户能够完成以下事情，则 1.0 成立：

- 本地部署并启动系统
- 注册登录
- 创建自己的族谱项目
- 持续保存和再次打开
- 编辑后关闭系统，下次还能继续
- 导出自己的数据，不被系统锁死
- 出现误操作时有恢复手段
- 愿意把它当成一个长期整理家人信息的工具，而不只是一次性尝试

## 6.2 第二阶段：2.0 系统化

### 目标

把“族谱编辑器”升级为“家族资料系统”。

### 阶段定义

2.0 的重点不再是单纯编辑谱图，而是构建真正的业务主数据体系，使系统可以长期管理家族资料。

### 核心能力

- 人物档案中心
- 亲属关系模型
- 婚姻、父母、子女、支系管理
- 家族事件管理
- 文档、照片、资料归档
- 数据来源与证据备注
- 搜索、筛选、浏览
- 后台维护页
- 前台查阅页
- 修订记录与基础审计
- 更成熟的家族记忆浏览入口
- 适合非维护者使用的浏览体验

### 技术重点

这一阶段最重要的不是页面数量，而是领域建模。

建议逐步建立真正的业务实体：

- Person
- Relationship
- Family
- Branch
- Generation or Lineage
- Event
- Attachment
- Revision
- User

此时，前端当前的“画布数据结构”应逐渐转型为：

- 一套用于可视化编辑和渲染的视图模型
- 而不是系统唯一的数据真相

真正的数据真相应逐步沉淀到后端业务模型中。

### 成功标准

如果系统不只是能画谱，而是能稳定地维护人物、关系、资料和修订记录，则 2.0 成立。

## 6.3 第三阶段：3.0 差异化

### 目标

做出与普通管理系统明显不同、具有长期壁垒的能力。

### 最值得投入的方向

- OCR 旧谱、碑文、扫描件
- 自动提取人物、关系、时间信息
- 发现重名、冲突、缺失关系
- 关系路径查询
- 时间轴展示
- 家族迁徙图
- 支系对照分析
- AI 辅助录入和校对

### 原则

AI 与高级可视化必须服务于“中文家族资料数字化”这个主题，不能为了功能数量牺牲系统聚焦度。

### 成功标准

当系统已经不仅是一个可本地部署的族谱产品，而是具备独特数字化整理能力的家族记忆工具链时，3.0 才算真正成立。

## 7. 架构演进建议

## 7.1 当前架构特点

当前项目已经形成前后端分离雏形：

- 前端：Vue 3 + TypeScript，可视化编辑器
- 后端：Spring Boot + MySQL，能力仍偏基础

但实际主数据仍在前端，后端暂未成为核心业务承载层。

## 7.2 推荐目标架构

未来建议演进为“双层能力结构”：

### A. 业务主数据层

由 Spring Boot + MySQL 承担：

- 用户与身份
- 族谱项目
- 人物主数据
- 关系主数据
- 事件资料
- 修订记录
- 附件元数据

### B. 可视化工作台层

由现有 Vue 前端继续承担：

- 族谱树编辑
- 关系可视化
- 导入导出
- 打印排版
- 交互式浏览

这样做的好处是：

- 保留当前项目最强的视觉表达能力
- 避免把前端重写成普通后台
- 逐步引入数据治理和长期存储能力

## 7.2.1 推荐产品层次

从产品信息架构上，建议逐步形成“两层入口”：

### A. 记忆层

面向大多数用户，强调：

- 家庭关系图
- 家人卡片
- 照片与故事
- 时间线

### B. 编辑层

面向维护者，强调：

- 关系编辑
- 人物资料编辑
- 导入导出
- 历史版本
- 项目设置
- 数据检查

这样可以避免所有用户一上来就面对“管理后台式”的产品感受。

## 7.3 关键设计原则

- 后端保存“业务真相”
- 前端维护“渲染与编辑所需的视图结构”
- 两者之间通过转换层衔接

不建议长期把前端当前 JSON 结构直接当成最终业务主模型。

## 8. 工程边界控制建议

为了降低工程压力，必须明确以下边界：

### 8.1 不要在 1.0 同时做完所有事情

如果在 1.0 就同时推进：

- 范式化数据库
- 完整后台
- 多人协作
- 审批流
- AI 能力
- 高级可视化

项目会很容易失控。

### 8.2 先做能闭环的最小系统

优先保证以下闭环：

- 登录
- 创建族谱
- 编辑
- 保存
- 再次打开
- 导出
- 恢复

只要这个闭环完成，产品基础就立住了。

### 8.3 不把项目做成通用 CMS

本项目的价值不在“页面多”或“管理项多”，而在：

- 可视化强
- 本地部署
- 数据归用户
- 场景聚焦中文族谱与家族资料

### 8.4 不把首页做得过重

如果首页一上来就是：

- 宗族
- 支系
- 世系
- 管理

会天然抬高新用户的心理门槛。

更推荐的首页语言与入口是：

- 家人
- 关系图
- 照片与故事
- 家庭档案

先让用户愿意进入，再逐步承接深层维护需求。

## 9. 当前最值得推进的近期任务

建议近期开发优先顺序如下：

1. 明确 1.0 的保存模型
2. 设计最小后端表结构
3. 建立族谱项目创建、读取、保存接口
4. 将前端编辑器从本地草稿升级为服务端持久化
5. 将上传文件与族谱项目、人物信息关联
6. 增加基础保存状态与恢复机制
7. 完善本地部署和启动说明

## 10. 项目最终愿景

本项目不应只停留在“做一个可视化好看的家谱页面”，而应逐步发展为：

**一套真正属于家族自己的、可本地部署的、可长期传承的家族记忆系统。**

如果继续沿着这条路线推进，它未来的价值将不只体现在代码规模上，而体现在以下几个方面：

- 回应真实需求，而不是空泛造轮子
- 兼顾产品体验与技术深度
- 兼顾视觉表达与数据治理
- 具备长期延展到 AI 辅助整理的可能性

它的价值不只在“记录”，更在于：

- 帮助今天的人保留来处
- 帮助以后的人重新看见关系与故事
- 在日益原子化的生活结构中，尽可能留下仍可连接过去的线索

## 11. 一句话结论

当前项目已经是一个方向明确、特色鲜明的作品。

下一步不应急于把它做成臃肿平台，而应先把它稳稳推进到一个真正可用的产品，再逐步成长为一套具有长期价值的家族记忆系统。
