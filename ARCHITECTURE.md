# 项目架构说明

更新日期：2026-04-26

## 1. 架构总览

当前项目采用前后端分离结构：

- 前端：[frontend](C:/Users/HX/Desktop/族谱管理系统/frontend)
- 后端：[backend](C:/Users/HX/Desktop/族谱管理系统/backend)

但从职责角度看，当前架构仍处于过渡态：

- 前端承担了几乎全部族谱编辑与主要数据流转逻辑
- 后端只承载基础认证与文件上传能力

所以当前系统是“结构上分离，能力上前端偏重”。

## 2. 当前前端架构

### 2.1 角色定位

前端目前不是普通展示页面，而是整个项目的核心工作台。

它主要负责：

- 谱图编辑
- 人物编辑
- 关系维护
- 布局计算
- 历史记录
- 草稿恢复
- 导入导出
- 打印与 SVG 输出

### 2.2 关键分层

前端当前大体可分为以下几层：

- `components/`
  负责界面组件与交互承载

- `composables/`
  负责编辑相关状态编排与界面逻辑复用

- `features/`
  负责相对独立的能力模块，如历史、持久化、校验、编辑操作

- `lib/`
  负责布局计算、辅助算法等底层逻辑

- `types/`
  负责数据结构定义

### 2.3 当前核心数据结构

前端当前族谱编辑最核心的数据结构来自 [family.ts](C:/Users/HX/Desktop/族谱管理系统/frontend/src/types/family.ts)：

- `Person`
- `FamilyUnit`
- `PublicationData`
- `PublicationSettings`

当前核心模型可以简化理解为：

- `people`：人物集合
- `families`：家庭单元集合
- `focusFamilyId`：当前聚焦家庭

这套结构很适合渲染和编辑谱图，但更偏视图模型，不应长期等同于后端业务主模型。

## 3. 当前后端架构

### 3.1 角色定位

后端目前承担的是轻量基础设施角色。

当前主要能力：

- 用户注册与登录
- 文件上传
- MySQL 连接

### 3.2 当前模块

- `controller/`
  对外接口层

- `service/`
  业务处理层

- `repository/`
  数据访问层

- `model/`
  实体定义层

- `dto/`
  接口收发数据结构

- `config/`
  配置层

### 3.3 当前主要接口

- `/api/auth/register`
- `/api/auth/login`
- `/api/upload`

目前还没有真正的族谱项目接口、人物接口、关系接口或资料管理接口。

## 4. 当前数据流特点

当前最关键的数据流特点如下：

### 4.1 认证数据流

- 前端通过 [auth.ts](C:/Users/HX/Desktop/族谱管理系统/frontend/src/api/auth.ts) 调用认证接口
- 登录成功后将 token 和用户名存入本地存储
- 前端用本地 token 决定是否进入工作台

### 4.2 族谱编辑数据流

- 前端加载样例数据或本地数据
- 用户在前端直接修改人物与关系
- 编辑历史在前端维护
- 草稿通过浏览器本地存储保存
- 导入导出由前端主导

### 4.3 上传数据流

- 前端上传头像文件到后端
- 后端写入本地 `uploads/`
- 后端返回静态访问地址
- 前端把返回的 URL 写入人物头像字段

## 5. 当前架构的优点

- 可视化体验强
- 编辑闭环清晰
- 本地工作流顺畅
- 前端能力已经具备明显辨识度
- 后端已具备作为后续扩展基础的雏形

## 6. 当前架构的限制

- 族谱主数据不在后端
- 没有项目级持久化
- 没有真正的多次会话连续性保障
- 当前后端还不是业务核心
- 当前数据结构更偏画布渲染，难以直接承载长期业务扩张

## 7. 推荐目标架构

未来建议演进为“双层架构”：

### 7.1 业务主数据层

由 Spring Boot + MySQL 承担：

- 用户
- 族谱项目
- 人物主数据
- 关系主数据
- 家族事件
- 附件信息
- 修订记录

### 7.2 可视化工作台层

由 Vue 前端继续承担：

- 谱图可视化编辑
- 关系交互
- 导入导出
- 打印排版
- 展示与浏览

## 8. 推荐的数据职责划分

建议未来逐步形成以下原则：

- 后端保存业务真相
- 前端维护渲染与交互所需的视图模型
- 两者之间通过转换层衔接

这意味着：

- 后端不应只存前端临时状态
- 前端也不应永久承担唯一数据真相

## 9. 推荐演进顺序

最稳的演进顺序建议为：

1. 先增加项目级持久化
2. 先把编辑内容保存到后端
3. 再引入项目、内容、附件等最小模型
4. 然后逐步过渡到人物、关系、事件等业务主模型

## 10. 架构结论

当前项目最重要的架构策略，不是推翻现有前端，而是：

**保留现有可视化工作台作为核心优势，再逐步补上后端主数据与产品底座。**
