## 3. 当前项目的简短判断

当前项目已经是一个较成熟的族谱可视化作品，但仍未完成产品化。

最关键的现实判断是：

- 前端强
- 后端轻
- 视觉体验成立
- 服务端持久化尚未打通

因此，后续开发时要始终记住：

**不要把它误判成“已经是完整管理系统”。**

## 4. 当前最重要的方向

当前最建议推进的不是“大扩张”，而是“先站稳产品底座”。

优先方向：

1. 项目级保存
2. 服务端持久化
3. 用户拥有自己的族谱项目
4. 上传与项目内容关联
5. 恢复与版本能力

## 5. 当前最推荐的 1.0 策略

1.0 阶段建议采用“轻量后端承载 + 保留前端编辑器核心”的策略。

重点是：

- 不推翻现有可视化工作台
- 不急着做完整后台体系
- 不急着完全范式化全部族谱关系
- 先把用户、项目、内容、附件四类数据站稳

## 6. 当前最需要警惕的坑

### 6.1 过早全面建模

如果在 1.0 就试图把人物、婚姻、支系、事件、证据、审核等全部彻底设计完，很容易被建模拖住。

### 6.2 把前端当前 JSON 直接视为最终模型

当前前端结构适合编辑和渲染，不适合作为长期唯一主模型。

### 6.3 一次性上太多后台功能

复杂权限、审批、多组织协作、AI、报表等都很有价值，但如果太早介入，会大幅增加复杂度。

## 7. 当前建议的近期任务

如果准备进入实际开发，最推荐的近期任务顺序是：

1. 明确 1.0 保存模型
2. 设计最小后端表
3. 增加族谱项目创建、读取、保存接口
4. 将前端本地草稿扩展为服务端保存
5. 补保存状态、错误提示、恢复逻辑
